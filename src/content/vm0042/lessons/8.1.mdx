<h2 class="text-2xl font-bold text-gray-800 mb-4">VMD0053: Model Eligibility, Calibration & the Project Domain</h2>

<HighlightBox>
  <p class="font-semibold text-green-800">When does VMD0053 apply?</p>
  <p class="text-green-900 mt-1">VM0042 Approach 1 allows the use of a <strong>biogeochemical model</strong> (e.g., DayCent, DNDC, RothC) to estimate SOC stock changes and GHG fluxes instead of relying entirely on field measurements. When you use such a model, you must follow <strong>VMD0053 v2.1</strong>, a binding methodology module that defines exactly how the model must be calibrated, validated, and its uncertainty quantified before any VCUs can be issued.</p>
</HighlightBox>

<AnalogyBox>
  <p class="font-semibold text-blue-800">🏋️ Analogy: The Certified Scale</p>
  <p class="mt-2 text-blue-900">Imagine a carbon project is a weighing station for carbon. If you claim to weigh 10,000 tonnes of CO₂ reduction, your scale (the biogeochemical model) needs to be <em>certified</em>, meaning independently tested, shown to give accurate readings under conditions similar to your project, and its measurement error must be documented. VMD0053 is the certification standard for that scale.</p>
</AnalogyBox>

<ExampleBox>
  <p class="font-semibold text-amber-800">📍 Which Models Are Actually Used in VM0042 Projects?</p>
  <ul class="mt-2 space-y-2 text-sm">
    <li><strong>DayCent (USA, Argentina, Brazil):</strong> Developed by Colorado State University, DayCent simulates daily soil C and N dynamics. It has peer-reviewed validation for corn, soy, and grassland systems and is the most commonly used model in Verra-registered IALM projects in the Americas.</li>
    <li><strong>RothC (UK, Australia, Europe):</strong> A simpler monthly-timestep model focused on SOC only (no N₂O or CH₄). Used for temperate cropland projects in Europe and Australia where N₂O is handled via Approach 3 default factors.</li>
    <li><strong>DNDC (Asia):</strong> Best validated for CH₄ from flooded rice systems and N₂O from intensive N fertilization, making it the go-to for rice AWD projects in Vietnam, China, and India.</li>
  </ul>
</ExampleBox>

<h3 class="text-xl font-bold mt-6 mb-3">Four Criteria a Model Must Meet (Section 4)</h3>
<p class="mb-3 text-sm">Any model used under VMD0053 must satisfy ALL FOUR of these criteria:</p>

<ResponsiveTable>
<table class="mb-6 text-sm">
  <tr><th>#</th><th>Criterion</th><th>What It Means in Practice</th></tr>
  <tr><td><strong>1</strong></td><td>Publicly available documentation</td><td>Model concept, inputs, outputs, and how it represents carbon dynamics must be publicly documented, source code is NOT required</td></tr>
  <tr><td><strong>2</strong></td><td>Peer-reviewed</td><td>Must have published, peer-reviewed studies showing it successfully simulates carbon stock and GHG changes from ALM practices</td></tr>
  <tr><td><strong>3</strong></td><td>Reproducible</td><td>Clear version control; same version used for baseline and project; all parameters reported; for stochastic models, random seed sequence provided</td></tr>
  <tr><td><strong>4</strong></td><td>Validated per this module</td><td>Model prediction error calculated from independent datasets for each PC/CFG/ES combination used in the project</td></tr>
</table>
</ResponsiveTable>

<div class="bg-amber-50 border border-amber-200 rounded p-3 mb-6 text-sm">
  <p class="font-semibold text-amber-800">Why Reproducibility Matters</p>
  <p class="mt-1 text-amber-900">A carbon credit is only credible if a verifier can run the model with the same inputs and get the same result. VMD0053 therefore requires: (1) a stable, version-controlled model, (2) the same version used in baseline and project scenarios, and (3) all internal parameters fully reported. For stochastic models that include randomness, the random seed sequence must be documented so the VVB can replicate exact outputs.</p>
</div>

<h3 class="text-xl font-bold mt-6 mb-3">Key Vocabulary: PC, CFG, and ES Combinations</h3>
<p class="mb-3 text-sm">VMD0053 validation must be demonstrated separately for every unique combination of:</p>

<div class="grid grid-cols-1 md:grid-cols-3 gap-4 mb-6">
  <div class="bg-blue-50 border border-blue-200 rounded p-4 text-sm">
    <h4 class="font-bold text-blue-800 mb-2">Practice Category (PC)</h4>
    <p class="text-blue-900 mb-2">The type of agricultural management change. Six PCs defined in Table 1:</p>
    <ol class="list-decimal ml-4 space-y-1 text-blue-900">
      <li>Inorganic nitrogen fertilizer application</li>
      <li>Organic amendments (biochar, compost, manure)</li>
      <li>Water management/irrigation/flooding</li>
      <li>Soil disturbance &amp; residue management (tillage)</li>
      <li>Land cultivation, planting &amp; harvesting (crop rotations, cover crops)</li>
      <li>Grazing practices</li>
    </ol>
    <p class="text-blue-800 text-xs mt-2"><strong>Domain of Practice Effects:</strong> Each PC must also be validated across its full "domain", e.g., for Inorganic N Fertilizer, validation must cover variations in magnitude, form, timing, and application method. Simply validating one rate of urea does not validate the full PC.</p>
  </div>
  <div class="bg-purple-50 border border-purple-200 rounded p-4 text-sm">
    <h4 class="font-bold text-purple-800 mb-2">Crop Functional Group (CFG)</h4>
    <p class="text-purple-900 mb-2">Groups crops by shared biological characteristics:</p>
    <ul class="space-y-1 text-purple-900">
      <li>• N-fixation (Y/N)</li>
      <li>• Annual vs. perennial</li>
      <li>• Photosynthetic pathway (C3/C4/CAM)</li>
      <li>• Growth form (tree/shrub/herbaceous)</li>
      <li>• Crop variety (e.g., rice variety)</li>
      <li>• Flooded vs. not flooded</li>
    </ul>
    <p class="text-purple-800 text-xs mt-2">Example: Maize = C4, annual, herbaceous, non-fixing, not flooded</p>
  </div>
  <div class="bg-green-50 border border-green-200 rounded p-4 text-sm">
    <h4 class="font-bold text-green-800 mb-2">Emission Source (ES)</h4>
    <p class="text-green-900 mb-2">The specific GHG flux being modeled:</p>
    <ul class="space-y-1 text-green-900">
      <li>• SOC stock change (CO₂)</li>
      <li>• N₂O flux</li>
      <li>• CH₄ flux</li>
    </ul>
    <p class="text-green-800 text-xs mt-2">Each ES requires separate validation. You cannot validate SOC change and then claim N₂O is also covered.</p>
  </div>
</div>

<ExampleBox>
  <p class="font-semibold text-amber-800">📐 Worked Example: How Many PC/CFG/ES Combinations?</p>
  <p class="mt-2 text-sm">A VM0042 project in Maharashtra introduces: (1) reduced tillage with maize, (2) cover cropping with legumes, and (3) reduced synthetic nitrogen fertilizer. The model must estimate SOC change and N₂O flux.</p>
  <ResponsiveTable>
<table class="text-sm mt-2">
    <tr><th>Practice Category</th><th>CFG</th><th>Emission Source</th><th>Must validate?</th></tr>
    <tr><td>Soil disturbance/residue (tillage)</td><td>C4 annual herbaceous (maize)</td><td>SOC change</td><td>✓ Yes</td></tr>
    <tr><td>Soil disturbance/residue (tillage)</td><td>C4 annual herbaceous (maize)</td><td>N₂O flux</td><td>✓ Yes</td></tr>
    <tr><td>Land cultivation/cropping (cover crop)</td><td>C3 annual herbaceous N-fixing (legume)</td><td>SOC change</td><td>✓ Yes</td></tr>
    <tr><td>Land cultivation/cropping (cover crop)</td><td>C3 annual herbaceous N-fixing (legume)</td><td>N₂O flux</td><td>✓ Yes</td></tr>
    <tr><td>Inorganic N fertilizer</td><td>C4 annual herbaceous (maize)</td><td>N₂O flux</td><td>✓ Yes</td></tr>
  </table>
</ResponsiveTable>
  <p class="mt-2 text-sm text-amber-800"><strong>Result:</strong> 5 separate PC/CFG/ES combinations must be validated before any credits can be issued from this model.</p>
</ExampleBox>

<h3 class="text-xl font-bold mt-6 mb-3">Model Calibration (Section 5.1)</h3>
<p class="mb-3 text-sm">Calibration adjusts model parameters to improve performance using a <em>training dataset</em>. VMD0053 does not prescribe a single calibration method but imposes strict requirements:</p>

<div class="grid grid-cols-1 md:grid-cols-2 gap-4 mb-6">
  <div class="bg-slate-50 border border-slate-200 rounded p-4 text-sm">
    <h4 class="font-bold text-slate-800 mb-2">Frequentist Calibration</h4>
    <ul class="space-y-1 text-slate-700">
      <li>• Adjust parameters using model sensitivities and overall performance</li>
      <li>• May use direct measurements (leaf area index, harvest index)</li>
      <li>• Same parameter sets used in validation AND in baseline/project simulations</li>
      <li>• Parameters defined at no finer than one IPCC climate zone OR nationally defined agricultural region</li>
      <li>• Calibration and validation datasets must be <em>demonstrably independent</em>, different experimental sites, no overlapping studies</li>
    </ul>
  </div>
  <div class="bg-indigo-50 border border-indigo-200 rounded p-4 text-sm">
    <h4 class="font-bold text-indigo-800 mb-2">Bayesian Calibration</h4>
    <ul class="space-y-1 text-indigo-900">
      <li>• Integrates prior knowledge + observed data via MCMC sampling</li>
      <li>• Uninformative priors: used when little prior knowledge exists → posterior driven by data</li>
      <li>• Informative priors: tight posterior closely follows the prior; data have less impact</li>
      <li>• Particularly recommended for uncertainty quantification, confidence intervals reflect data availability</li>
      <li>• Sparse data = wide confidence intervals (conservative for crediting)</li>
    </ul>
  </div>
</div>

<div class="bg-amber-50 border border-amber-200 rounded p-3 mb-6 text-sm">
  <p class="font-semibold text-amber-800">Critical Rule: Same Parameters Throughout</p>
  <p class="mt-1 text-amber-900">The parameter sets used in calibration (and confirmed in validation) must be the SAME sets used when calculating baseline and project GHG estimates. You cannot "tune" the model for better performance in validation and then use different parameters in the actual project calculations. This ensures what was tested is what is used.</p>
</div>

<h3 class="text-xl font-bold mt-6 mb-3">Defining the Project Domain (Section 5.2.1–5.2.3)</h3>
<p class="mb-3 text-sm">The project domain is the biophysical space within which the model has been validated. It has three dimensions:</p>

<ResponsiveTable>
<table class="mb-6 text-sm">
  <tr><th>Dimension</th><th>How Defined</th><th>Example (India project)</th></tr>
  <tr><td><strong>CFGs</strong></td><td>All crop functional groups grown in the project</td><td>C4 annual (maize), C3 annual N-fixing (soybean), C3 annual (wheat)</td></tr>
  <tr><td><strong>Climate Zones</strong></td><td>2019 IPCC Refinements classification OR nationally defined regions</td><td>Warm temperate moist + warm temperate dry (two zones in project)</td></tr>
  <tr><td><strong>Soils</strong></td><td>Soil texture class (FAO/USDA triangle) + clay content range</td><td>Clay loam and loam; clay content 25–40%</td></tr>
</table>
</ResponsiveTable>

<p class="mb-3 text-sm"><strong>Validation dataset requirements for the project domain:</strong></p>
<ul class="list-disc ml-6 text-sm space-y-1 mb-4">
  <li>At minimum, every climate zone in the project must be represented in the validation dataset</li>
  <li>The three most predominant soil textural classes must be covered</li>
  <li>Clay content in the dataset must span at least <strong>15 percentage points</strong></li>
  <li>Priority: select studies geographically closest to the project location</li>
  <li>Studies must report location, management, starting soil conditions, and all model inputs</li>
</ul>

<h3 class="text-xl font-bold mt-6 mb-3">Substitution for Missing Crop Types (Section 5.3)</h3>
<p class="mb-3 text-sm">What if no validation data exist for a specific crop in the project? VMD0053 permits substitution under a two-tier hierarchy:</p>
<div class="bg-amber-50 border border-amber-200 rounded p-3 mb-3 text-sm">
  <p class="font-semibold text-amber-800">Hierarchy: Same CFG first, then different CFG</p>
  <p class="mt-1 text-amber-900">If no data exist for a specific crop, first try to substitute with another validated crop <em>within the same CFG</em> (e.g., substitute one C4 annual non-N-fixing crop for another). Only if that is not possible should you substitute with a crop from a different CFG.</p>
</div>
<div class="grid grid-cols-1 md:grid-cols-2 gap-4 mb-6">
  <div class="bg-blue-50 border border-blue-200 rounded p-3 text-sm">
    <h4 class="font-bold text-blue-800 mb-1">Baseline Substitution</h4>
    <p class="text-blue-900">Replace missing crop with a validated CFG that emits <em>FEWER</em> GHGs than the missing crop. This is conservative because it underestimates the baseline → fewer credits claimed.</p>
  </div>
  <div class="bg-green-50 border border-green-200 rounded p-3 text-sm">
    <h4 class="font-bold text-green-800 mb-1">Project Substitution</h4>
    <p class="text-green-900">Replace missing crop with a validated CFG that emits <em>MORE</em> GHGs than the missing crop. This is conservative because it underestimates the project benefit → fewer credits claimed.</p>
  </div>
</div>
<p class="text-sm text-slate-600 mb-4">Both substitutions require peer-reviewed literature support. If no conservative alternative exists, substitution is not permitted and the model cannot be used for that CFG.</p>

<div class="bg-green-50 border border-green-200 rounded p-4 text-sm">
  <p class="font-semibold text-green-800">Key Takeaways: Section 5.1 & Domain Definition</p>
  <ul class="mt-2 space-y-2 text-green-900">
    <li>✓ Four eligibility criteria must ALL be met before a model can be used</li>
    <li>✓ Every PC/CFG/ES combination used in the project must be separately validated</li>
    <li>✓ Parameters used in calibration = parameters used in project simulations (no switching)</li>
    <li>✓ The project domain must be covered by the validation dataset (climate zones, soil textures, clay range)</li>
    <li>✓ Substitution for missing crops is permitted only if it is conservative in both directions</li>
  </ul>
</div>
